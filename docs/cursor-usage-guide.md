## 🤖 模型选型与工具对比

### 🌍 Agent 工具生态

目前主流的 AI Agent 工具包括 Claude Code（Anthropic 官方 CLI）、OpenClaw（开源 Agent 框架）、Cursor 等，均提供 Agent 模式。各工具的核心差异不在交互界面，而在于 **底层调用的大语言模型能力**。

### 📊 Cursor 可用模型

| 模型 | 厂商 | 特点 |
|:-----|:-----|:-----|
| ⭐ **Claude Opus 4.6** | Anthropic | 当前综合能力最强，代码生成、多步推理、长上下文理解均表现优异 |
| GPT-5.3 Codex | OpenAI | 代码生成能力突出，复杂推理场景略逊于 Opus |
| GPT-5.4 | OpenAI | OpenAI 最新旗舰模型，综合能力接近 Opus |

### 💰 成本与效果分析

**❌ 直接调用 API 的问题：**

- **成本高昂**：Opus 4.6 API 定价约 $15/M input + $75/M output，一个复杂任务多轮对话可达数美元
- **集成成本**：需自行搭建 Agent 框架、管理上下文窗口、处理工具调用协议，开发维护成本不低

**⚠️ 国产模型的局限：**

在 Agent 场景（多步推理 + 工具调用 + 代码生成）中，国产模型与 Opus 仍存在明显差距。实际测试中，使用 OpenClaw + Kimi K2.5 / K2.5 Code 执行邮件发送任务时，模型能正确生成工具调用语法，但无法完成实际执行，最终仍需借助 Cursor + Opus 进行问题排查和修复。该类问题在涉及多工具协调的复杂工作流中尤为突出。

**✅ 推荐方案：**

Cursor Pro 订阅（$20/月）已包含充足的高端模型调用额度，无需单独购买 API。

> 💎 **建议直接使用 Cursor + Claude Opus 4.6 组合，兼顾能力上限与使用成本。**

### 🎛️ 四种交互模式

Cursor 提供四种模式，覆盖不同使用场景：

| 模式 | 图标 | 权限 | 定位 | 适用场景 |
|:-----|:----:|:-----|:-----|:---------|
| **Agent** | 🤖 | 读写文件、执行命令、安装依赖 | 自主执行，端到端完成任务 | 功能开发、项目搭建、代码重构 |
| **Plan** | 📝 | 只读分析，确认后再执行 | 先规划后执行，降低风险 | 架构设计、大型改动、技术选型 |
| **Debug** | 🔍 | 聚焦问题排查与错误上下文 | 定向诊断与修复 | Bug 定位、报错分析、性能排查 |
| **Ask** | 💬 | 只读，不修改任何文件 | 纯问答与代码分析 | 理解代码逻辑、技术咨询、学习 |

**🎯 模式选择速查：**

- 🤖 **日常开发 → Agent**：大多数任务可由 AI 自主完成
- 📝 **重大改动 → Plan**：先审查方案再决定是否执行
- 🔍 **遇到报错 → Debug**：聚焦错误上下文，快速排查
- 💬 **纯粹问问 → Ask**：避免 AI 误修改文件

## ⚔️ 关键实践：对抗式审查（像 GAN 一样迭代）

> 🧠 **核心思想：** 不要让 AI 既当运动员又当裁判。每一个方案产出后，都应该让 AI 切换身份进行审查和挑战，通过"生成 → 对抗 → 修正"的循环逐步提升质量。

### 为什么需要对抗式审查

AI 生成的方案往往"看起来很完整"，但可能存在：

- 🎯 方向偏差 —— 理解了你的话，但没理解你的意图
- 🕳️ 逻辑漏洞 —— 表面自洽，细节经不起推敲
- 📦 过度设计 —— 简单问题复杂化，引入不必要的依赖
- 🔒 安全隐患 —— 硬编码密钥、缺少异常处理等

如果直接采纳第一版输出，这些问题会被带进最终产物。

### 操作方法

**第一步：让 AI 生成方案（生成器 G）**

正常使用 Agent 或 Plan 模式完成任务。

**第二步：让 AI 切换身份审查（判别器 D）**

生成完成后，给 AI 一个新的角色：

> "现在你是一个资深的技术评审，请审查上面的方案，指出不合理的地方、潜在风险和改进建议。"

或者更具针对性：

> "假设你是安全工程师，审查这段代码的安全风险。"
>
> "假设你是用户，从使用体验角度评价这个设计。"
>
> "假设你是测试工程师，列出这个方案需要覆盖的测试用例。"

**第三步：根据审查结果修正（迭代优化）**

AI 会指出具体问题，然后你可以让它基于审查意见进行修改。这个过程可以多轮迭代，直到方案质量达标。

### 迭代流程

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 🤖 生成   │────→│ ⚔️ 审查   │────→│ 🔧 修正   │
│ (G: 生成器)│     │ (D: 判别器)│     │ (改进方案) │
└──────────┘     └──────────┘     └─────┬────┘
                                        │
      ┌─────────────────────────────────┘
      │  不满意则继续迭代
      ▼
┌──────────┐     ┌──────────┐
│ 🤖 再生成  │────→│ ⚔️ 再审查  │────→ ✅ 达标，采纳
└──────────┘     └──────────┘
```

### 实际效果

| 场景 | 无审查 | 有对抗式审查 |
|:-----|:-------|:-------------|
| 代码方案 | 可能遗漏边界条件 | 审查后补全异常处理和边界 case |
| 架构设计 | 可能过度复杂 | 审查后精简不必要的抽象层 |
| 文档撰写 | 可能遗漏关键步骤 | 审查后补充前置条件和注意事项 |
| 脚本开发 | 可能硬编码配置 | 审查后抽取为配置文件或环境变量 |

> 💡 **一句话总结：生成一版不够好，让 AI 自己"找茬"，再改一版 —— 像 GAN 的生成器和判别器一样对抗迭代，质量会大幅提升。**
